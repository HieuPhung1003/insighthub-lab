## Triệu chứng

Toàn bộ request tới `POST /chat` trả về **HTTP 500**. Người dùng không thể
đặt câu hỏi qua giao diện RAG. Các endpoint khác (`/healthz`, `/documents`)
không bị ảnh hưởng.

### Histogram bucket — smoking gun

```
insighthub_rag_query_latency_seconds_bucket:
  le=0.1  → 0    le=0.25 → 0    le=0.5  → 0
  le=1.0  → 0    le=2.5  → 0    le=5.0  → 0
```

Observation duy nhất nằm hoàn toàn ngoài bucket 30 s, xác nhận request bị
block chính xác bằng `pool timeout = 30s`.

## Nguyên nhân gốc

### Nguyên nhân trực tiếp

`DATABASE_URL` trong Kubernetes Secret `insighthub-db` chứa password có **ký
tự URL-reserved không được percent-encode**:

```
password: 4>OZ]0Hr7rM#<cl%!06x_lfw
                ^  ^  ^
                |  |  └── '#' được URL parser hiểu là fragment delimiter
                |  └───── ']' làm parser nhầm IPv6 address
                └──────── '>' là HTML/URL reserved char
```

psycopg3 parse `DATABASE_URL` dưới dạng **connection URI**. Khi gặp `#`, nó
bị cắt phần netloc — khiến hostname/credentials bị parse sai. Pool retry liên
tục trong 30 giây rồi raise `PoolTimeout`.

**API log (lặp liên tục):**
```
WARNING: psycopg.pool: error connecting in 'pool-1':
         invalid percent-encoded token: "4>OZ]0Hr7rM#<cl%!06x_lfw"
psycopg_pool.PoolTimeout: couldn't get a connection after 30.00 sec
```

### Nguyên nhân thứ cấp (phát hiện sau fix 1)

Schema RDS trống — `CREATE EXTENSION vector` chưa được chạy trên RDS instance.
Điều này không được phát hiện trước vì lỗi URL encoding che khuất hoàn toàn.

### Chuỗi lỗi đầy đủ

```
POST /chat
  └→ with rag_query_latency.time():      ← histogram bắt đầu tính giờ
       └→ retrieve(question)
            └→ pool.getconn(timeout=30)  ← block 30s
                 └→ urlparse(DATABASE_URL) → FAIL mỗi attempt
                 └→ PoolTimeout raised
  └→ rag_query_latency records 30.0 s   ← context manager exit on exception
  └→ llm_call_latency: NOT reached      ← count stays 0
  └→ http_requests_total{status=500}++
  └→ HTTP 500 returned to user
```

---

## Fix đã thực hiện

### Fix 1 — Percent-encode DATABASE_URL password

```bash
# Trước
postgresql://insighthub:4>OZ]0Hr7rM#<cl%!06x_lfw@host:5432/insighthub

# Sau (python: urllib.parse.quote(password, safe=''))
postgresql://insighthub:4%3EOZ%5D0Hr7rM%23%3Ccl%25%2106x_lfw@host:5432/insighthub
```

Patch secret:
```bash
kubectl patch secret insighthub-db -n insighthub-dev \
  --type=json -p='[{"op":"replace","path":"/data/DATABASE_URL","value":"<base64>"}]'
```

### Fix 2 — Khởi tạo schema RDS

```bash
# Chạy init.sql thông qua kubectl exec
CREATE EXTENSION IF NOT EXISTS vector;   -- pgvector 0.8.1 available, not installed
CREATE TABLE IF NOT EXISTS documents (…);
CREATE TABLE IF NOT EXISTS chunks (embedding VECTOR(1024), …);
CREATE INDEX … USING hnsw …;
```

### Kết quả sau fix

| Endpoint | Trước fix | Sau fix |
|---|---|---|
| `GET /documents` | 500 — 30 s | **200 — 0.08 s** |
| `POST /chat` | 500 — 30 s | **404 — 0.05 s** (đúng: chưa có tài liệu) |
| Pool warnings | `invalid percent-encoded token` liên tục | **Sạch** |

## Đề xuất phòng ngừa

| # | Đề xuất | Ưu tiên |
|---|---|---|
| 1 | **Tách DB credentials thành env vars riêng** (`DB_HOST`, `DB_USER`, `DB_PASSWORD`) thay vì một URL string — tránh hoàn toàn vấn đề URL encoding | High |
| 2 | **Dùng AWS Secrets Manager + External Secrets Operator** để inject secret; password không bao giờ đi qua URL parsing | High |
| 3 | **Startup health check kết nối DB** — fail-fast với error rõ ràng thay vì để pool timeout 30 s mỗi request | Medium |
| 4 | **Chạy init.sql như Kubernetes Job** (không phải manual) khi deploy lên môi trường mới — đảm bảo schema luôn sẵn sàng | Medium |
| 5 | **Giới hạn ký tự trong RDS password**: chỉ dùng `[A-Za-z0-9_-]` khi tạo password để tránh URL encoding issue về sau | Low |
| 6 | **Alert on pool error rate**: thêm Prometheus counter cho `psycopg_pool.PoolTimeout` để phát hiện nhanh hơn | Low |

## Lessons Learned

1. **Histogram `+Inf` bucket là tín hiệu mạnh** — observation duy nhất nằm ngoài
   bucket max chỉ thẳng vào pool timeout, không phải LLM chậm hay mạng bị nghẽn.

2. **LLM call count = 0 là key differentiator** — nếu lỗi ở LLM layer, count sẽ > 0.
   Count = 0 chứng minh failure xảy ra tại tầng retrieval (DB), trước khi reach LLM.

3. **TCP reachable ≠ DB usable** — mạng và Security Group hoàn toàn OK, nhưng lớp
   URL parsing của client library là điểm thất bại. Luôn test end-to-end connection,
   không chỉ TCP.

4. **Một lỗi có thể che khuất lỗi khác** — URL encoding bug che hoàn toàn schema
   missing. Fix lần lượt từng layer là bắt buộc.
